The Problem

Your AI agents have no idea what your policies say.

GitHub Copilot, Cursor, Claude Code, Windsurf — each one operates in isolation. They don't know your IAM policy. They don't know which services are off-limits. They can't see your security runbook. And when they write code that creates an S3 bucket, modifies a schema, or calls a cloud API, there's no gate stopping them.

🔑

Capability Sprawl

Agents request IAM roles, write secrets, and modify infrastructure without any pre-flight check against your policy.

🕵️

Zero Provenance

When an incident hits, you have no way to trace which AI agent wrote the code, under what context, or with what declared intent.

📊

Drift Blindness

What agents declare they need and what they actually call in production are two different things — and no one is watching the gap.

The VertaAI Platform

Three tracks. One source of truth.

VertaAI intercepts the AI coding workflow at three distinct moments — giving your team control without slowing developers down.

Track 0
Pre-flight Permission Envelope
Before any code is written, the agent receives a permission envelope compiled from your active Policy Packs. Blocked capabilities are off the table from keystroke one.
  • Compiled from workspace Policy Packs
  • Injected into every editor's config files
  • 5 blocked + 7 declaration-required capabilities
  • Session budgets (max files, max abstractions)
Track A
YAML Policy Gate at PR Review
At pull request time, VertaAI runs 5 automated comparators against the PR context, declared intent, and policy packs. Results post as a GitHub Check.
  • Intent ↔ capability parity check
  • Duplicate abstraction risk scoring
  • Churn + complexity threshold enforcement
  • Over-permissioned import detection
Track B
Runtime Drift Detection
In production, VertaAI continuously monitors CloudTrail, GCP Audit Logs, and DB query logs for capability usage that was never declared in the intent artifact.
  • AWS CloudTrail + GCP Audit Log ingestion
  • Decay-weighted severity scoring
  • PagerDuty + Slack alerting
  • Auto-close when drift resolves

How It Works

From intent declaration to runtime audit — in three enforced steps.

Every AI-generated code change flows through the VertaAI governance pipeline. Here's what happens at each stage.

0
Track 0 · Pre-flight

Permission envelope compiled and injected

Before a developer opens their editor, VertaAI compiles a permission envelope from all active Policy Packs in the workspace. The envelope is delivered to Claude Code (via CLAUDE.md), GitHub Copilot (via copilot-instructions.md), Cursor (via .cursor/rules/vertaai-permissions.mdc), Windsurf (via .windsurfrules), and Augment (via .augment/settings.json) — automatically, on setup and on every policy change.

.cursor/rules/vertaai-permissions.mdc
--- alwaysApply: true --- # VertaAI Agent Permission Envelope # Compiled from workspace policy packs · Auto-updated 🚫 BLOCKED (do not generate code using these capabilities): iam_modify · secret_write · db_admin · infra_delete · deployment_modify ⚠️ DECLARE BEFORE USE (add to intent artifact): s3_delete · s3_write · schema_modify · network_public infra_create · infra_modify · secret_read 📋 SESSION BUDGETS: Max files changed per session: 20 Max new abstractions per session: 3
A
Track A · PR Gate

YAML policy evaluated at pull request time

When a PR is opened, VertaAI's policy evaluator runs five comparators against the code changes, the declared intent artifact, and your active Policy Packs. Results post as a GitHub Check that can block merge.

GitHub Check · VertaAI Policy Gate
✓ INTENT_CAPABILITY_PARITY Declared: [s3_write, schema_modify] Detected: [s3_write, schema_modify] Undeclared: none · PASS ✓ CHURN_COMPLEXITY_RISK Lines changed: 847 Complexity delta: +12 Threshold (pack: security-baseline): 1200 / +25 · PASS ⚠ DUPLICATE_ABSTRACTION_RISK Similarity score: 0.78 with UserAuthService.ts Risk: medium · flagged for review (non-blocking) ✓ OVERPERMISSIONED_IMPORT ✓ INFRA_OWNERSHIP_PARITY Result: PASS · Merge allowed
B
Track B · Runtime

Continuous capability drift detection in production

Post-deploy, VertaAI ingests AWS CloudTrail events, GCP Audit Logs, and database query logs. Any capability usage that wasn't declared in the intent artifact surfaces as a DriftCluster with decay-weighted severity scoring.

Provenance Debugging

Trace every line back to the agent that wrote it.

VertaAI's VSCode/Cursor CodeLens shows the AI author, PR number, and governance quality score inline — right above the function. When an incident hits, you know exactly who (or what) wrote the code.

userService.ts — CodeLens (VertaAI)
🤖 agent · PR #142 | Quality: 84/100 [view governance report] export async function createUser(data: CreateUserInput) { const user = await db.user.create({ data }); await mailer.sendWelcome(user.email); return user; }

Quality score is a 0–100 composite of 5 governance dimensions: abstraction safety, test coverage intent, churn risk, import hygiene, and capability declaration accuracy.

Policy Pack Overrides

Operators configure everything through Policy Packs.

The Agent Policy tab in the Policy Pack wizard lets operators add additional blocked capabilities, require human approval for specific ones, and set session budgets — all through a UI with no YAML hand-editing required. The most-restrictive policy wins across all active packs.

Policy Pack YAML · security-baseline
agentPolicy: additionalBlocked: - s3_write # elevated to blocked for this pack - infra_create requireApproval: - schema_modify # needs human sign-off before merge sessionBudgets: maxFilesChanged: 15 # overrides workspace default of 20 maxNewAbstractions: 2 # overrides workspace default of 3
